# Part 2: Welcoming a guest

`hasha` announces an adapted payload. `hasho` closes it. Between them, the material sounds like Phi but remains a visitor.

## Shape

A guest token must begin with one of Phi's onsets, use only permitted consonants, fricative digraphs, and vowels, keep every syllable open, avoid three adjacent vowels, remain lowercase, and take penultimate stress. Several adapted tokens may share one frame.

```
mia ne hasha kioto hasho lothea.
1SG NAME EXT.GUEST [kioto] EXT.GUEST.CLOSE love.
(I love Kyoto.)

lo mia hasha solarapanaka hasho shareo.
PL 1SG EXT.GUEST [solarapanaka] EXT.GUEST.CLOSE discuss.
(We discuss solarpunk.)
```

Repeated syllables are permitted here even though the core lexicon forbids them. `hasha mamama hasho` is structurally possible because listeners already know that `mamama` is payload, not a candidate core word.

## Adaptation is social

Phi provides constraints, not an automatic transliteration algorithm. Ask a name-bearer which approximation they accept. Preserve the source in an exact record when pronunciation loses a meaningful contrast. If no responsible adaptation is available, use exact form.

## The closer collision

Standalone `hasho` ends the frame. An adaptation that would contain it as its own token must change. This is the one sequence a guest payload cannot preserve; exact form remains available when changing it would be unacceptable.
