# Quick reference

| Question | Guest | Exact |
|---|---|---|
| Boundary | `hasha … hasho` | `patha … patho` |
| Payload | Phi-pronounceable adaptation | arbitrary opaque source material |
| Core lexicon lookup | no | no |
| Lowercase and period-only | yes | no |
| Tengwar rendering | full payload | Phi boundaries only; payload retained |
| Repeated syllable | allowed | irrelevant |
| Literal closer | re-adapt instead | double `patho patho` |
| Bare later mention | never | never |
| Automatic core promotion | never | never |

Both frames are nominal atoms. Put `ne`, prepositions, particles, and support verbs outside. Use core composition when its analysis is useful, guest form when Phi pronunciation matters, and exact form when fidelity matters. Marked carriage is not core adoption, approval, or condemnation.

For a proper name, first consider a productive onym: `ne` plus one lowercase, two- or three-syllable Phi-shaped token accepted by the bearer. It is a name atom rather than a guest frame and receives no lexicon meaning. Use guest boundaries for a longer or multi-token adaptation and exact boundaries for source-faithful form.

The validator checks guest shape and both boundaries. It ignores exact payload internally, then resumes every core rule at the closer. Generated HTML escapes payload before display.
